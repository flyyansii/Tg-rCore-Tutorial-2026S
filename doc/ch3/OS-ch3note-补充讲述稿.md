# rCore ch3 补充讲述稿：把“多个程序轮流运行”讲给别人听

如果要把 ch3 讲给一个刚学完计组和数据结构的人，我会先从一个问题开始：

为什么 ch2 不能算真正的多任务操作系统？

因为 ch2 虽然能运行多个程序，但它们是串行的。程序 A 结束后，内核才加载程序 B。中间没有“保存 A 的现场，再去跑 B，然后以后还能回到 A”的过程。

ch3 要补上的就是这个能力。

## 1. 从批处理到多任务

ch2 的系统像这样：

```text
运行 app0 -> app0 exit -> 运行 app1 -> app1 exit -> 运行 app2
```

ch3 的系统像这样：

```text
运行 app0 一会儿
切到 app1 一会儿
切到 app2 一会儿
再回到 app0
```

CPU 同一时刻其实只能执行一个程序，但切换很快，人就感觉多个程序在一起运行。这就是分时多任务。

## 2. 为什么必须保存上下文

一个程序被暂停时，不能只记住“它暂停了”。还要记住暂停时 CPU 寄存器的状态，例如：

```text
pc/sepc: 执行到哪条指令
sp: 当前栈在哪里
a0-a7: 函数参数、返回值和 syscall 参数
ra: 函数返回地址
其他通用寄存器: 当前计算的中间值
```

如果不保存这些，下一次恢复程序时，它就像失忆一样，不知道自己刚才算到哪里。

所以本章的本质是：

```text
暂停一个程序 = 保存它的上下文
恢复一个程序 = 恢复它的上下文
```

## 3. TCB 为什么重要

TCB 是 Task Control Block，任务控制块。可以把它理解成内核为每个程序建立的“档案袋”。

用户程序本身只知道自己要运行，但内核要管理它，就必须额外维护：

```text
这个任务的寄存器状态
这个任务的栈
这个任务是否结束
这个任务的系统调用统计
```

这就是 `TaskControlBlock`。如果没有 TCB，内核就没有地方存放“每个任务自己的状态”，也就无法做多任务。

## 4. 系统调用和调度的关系

用户程序不能直接操作内核数据结构，所以它如果想退出、输出、让出 CPU，就必须通过 `ecall` 进入内核。

例如普通输出：

```text
println!
  -> user_lib::write
  -> syscall
  -> ecall
  -> Trap 到内核
  -> 内核分发到 sys_write
```

`yield` 也类似：

```text
user_lib::sched_yield
  -> syscall
  -> ecall
  -> 内核识别 SCHED_YIELD
  -> 当前任务让出 CPU
  -> 调度下一个任务
```

所以系统调用不只是“调用内核函数”，它还是用户态和内核态之间的受控桥梁。

## 5. 时钟中断为什么是分时系统的灵魂

如果只有 `yield`，程序必须主动让出 CPU。但如果程序写死循环：

```rust
loop {}
```

它永远不 `yield`，整个系统就会被它卡死。

时钟中断解决这个问题。内核提前设置一个时间点，时间到了 CPU 自动打断当前用户程序，进入内核。内核就有机会说：

```text
这个程序时间片到了，换下一个。
```

这就是抢占式调度。

## 6. trace 练习怎么讲

`sys_trace` 是本章基础练习，它训练的是两个能力：

1. 如何在系统调用路径上记录信息。
2. 如何通过当前任务的 TCB 保存每个任务自己的统计数据。

它的三个功能是：

```text
trace(0, addr, _): 读当前任务 addr 地址的一个字节
trace(1, addr, data): 写当前任务 addr 地址的一个字节
trace(2, id, _): 查询当前任务调用 id 号 syscall 的次数
```

这里的“不安全”正好说明 ch4 的必要性：没有页表和地址空间之前，内核很难判断用户给的地址是否合法。

## 7. ch3-snake 怎么讲

贪吃蛇扩展实验的意义，不是“写一个小游戏”这么简单。它把本章几个概念串起来了：

```text
用户态游戏循环
  -> 需要 sleep 控制帧率
  -> 需要 read 获取键盘输入
  -> 需要 write 刷新画面
  -> 这些都通过 syscall 请求内核
```

这次实现里，游戏逻辑在用户态：

```text
ch3_snake.rs:
  蛇的位置
  食物位置
  移动方向
  撞到自己后的重置
  打包 SnakeFrame
```

内核只提供机制：

```text
read(STDIN): 从键盘设备拿一个按键
write(fd=3): 把 SnakeFrame 画到 VirtIO-GPU framebuffer
sleep: 控制游戏帧率
timer/yield: 保证任务调度继续运转
```

这比把游戏直接写进内核更符合 OS 的边界：内核提供机制，应用实现策略。

## 8. 为什么不用终端 ASCII 作为最终方案

终端版贪吃蛇实现简单，但有几个问题：

```text
画面会有残影
键盘输入和串口输出混在一起
不像老师 demo 的图形链路
无法体现 framebuffer / GPU 设备抽象
```

所以最终采用图形链路：

```text
用户态 write(fd=3, SnakeFrame)
  -> 内核识别图形帧
  -> VirtIO-GPU 写 framebuffer
  -> QEMU GTK 窗口显示
```

键盘也从终端 UART 改成了 `virtio-keyboard-device`，这样按键焦点在 QEMU 图形窗口里，用户体验更接近真正的小游戏。

## 9. 为什么要拆成 snake 和 snake-ci

这是工程上很重要的一点。

交互版：

```text
cargo run --features snake
```

它应该一直运行，等用户按键。否则游戏没法玩。

CI 版：

```text
cargo run --features snake-ci
```

它必须自动跑完并退出。否则 GitHub Actions 会一直等待，最后超时失败。

所以拆分不是偷懒，而是为了同时满足：

```text
人可以玩
机器可以测
```

## 10. 我现在讲 ch3 的一句话版本

ch3 做的事情是：内核把多个用户程序同时放进内存，为每个程序建立 TCB 保存执行现场，然后通过系统调用和时钟中断不断在程序之间切换，从而实现最小的分时多任务系统；在此基础上，用户态贪吃蛇通过 `read` 获取键盘输入，通过 `write(fd=3)` 请求内核把画面刷到 VirtIO-GPU framebuffer。

这句话里包含本章最关键的六个词：

```text
多程序加载
TCB
系统调用
时钟中断
VirtIO-GPU
VirtIO Keyboard
```
