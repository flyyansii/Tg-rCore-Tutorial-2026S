# rCore ch3 补充讲述稿：把“多任务系统”讲给别人听

如果我要把 ch3 讲给一个和我一样只学过计组、数据结构、C/嵌入式基础，但还没系统学过 OS 的同学，我不会一上来就讲 Rust 代码。我会先问一个问题：

> 为什么 ch2 能跑多个程序，但还不能叫真正的多任务？

因为 ch2 只是“一个接一个执行”。它像一个自动批改脚本：app0 结束后运行 app1，app1 结束后运行 app2。中间没有暂停、保存现场、恢复现场的能力。

ch3 补上的能力就是：程序 A 没结束，也可以先停下来，保存它的状态，然后去跑程序 B；以后还能回来继续跑程序 A。

这就是多任务系统的核心。

## 1. 用生活类比理解 ch3

可以把 CPU 想成一个人，这个人同一时间只能做一件事。

ch2 像这样：

```text
做完作业 A
再做作业 B
再做作业 C
```

ch3 像这样：

```text
作业 A 写 5 分钟，夹书签
作业 B 写 5 分钟，夹书签
作业 C 写 5 分钟，夹书签
回到作业 A，从书签处继续
```

这里“书签”就是上下文。对程序来说，上下文不是纸质书签，而是寄存器、栈指针、程序计数器等状态。

## 2. 多任务不是魔法，是快速切换

我一开始会以为“多个程序同时运行”就是它们真的同时在 CPU 上执行。后来发现对单核来说不是这样。

单核 CPU 同一时刻只能执行一个任务。多任务的本质是：

```text
保存当前任务状态
选择下一个任务
恢复下一个任务状态
让它继续跑
```

这个过程重复得足够快，用户就感觉多个程序像同时运行。

这也是为什么操作系统需要调度器。调度器的工作不是“让 CPU 同时执行多个任务”，而是决定“下一个时间片给谁”。

## 3. 为什么 TCB 是本章的核心

要管理一个任务，内核不能只知道“这个程序叫 app0”。内核必须知道：

```text
它当前执行到哪里
它的寄存器是什么状态
它的栈在哪里
它是否已经结束
它调用过哪些系统调用
```

这些信息放在 `TaskControlBlock` 里。

所以我讲 TCB 时会这样说：

> TCB 是任务在内核里的身份证和档案袋。用户程序是“人”，TCB 是内核给这个人建立的档案。

没有 TCB，内核就无法知道每个任务自己的状态，也无法在多个任务之间切换。

## 4. yield 和 exit 的区别怎么讲

`yield` 和 `exit` 都会让 CPU 从当前用户程序回到内核，但语义完全不同。

```text
yield:
  我还没结束，只是这轮先让别人跑。

exit:
  我已经结束，以后不用再调度我。
```

用学习类比：

```text
yield:
  我这道题先放一放，待会继续写。

exit:
  这张卷子写完了，可以交了。
```

在代码中，`yield` 返回的是调度事件，内核会把当前任务保留；`exit` 会让内核把任务标记为完成。

## 5. 为什么需要时钟中断

只靠 `yield` 的系统有个很大的缺点：它要求程序主动让出 CPU。

如果程序很乖：

```text
算一会儿 -> yield -> 算一会儿 -> yield
```

系统能正常运转。

但如果程序不乖：

```rust
loop {}
```

它永远不 yield，其他任务就没机会运行。

所以内核需要时钟中断。时钟中断相当于操作系统手里的闹钟。时间到了，CPU 不管当前用户程序愿不愿意，都必须 Trap 回内核。

这就是我理解的抢占：

```text
不是用户主动交出 CPU，
而是内核通过时钟把 CPU 收回来。
```

## 6. 系统调用为什么是“桥”

用户态和内核态之间不能随便跳。用户程序不能直接调用内核任意函数，也不能直接访问硬件。

系统调用就是一座受控的桥：

```text
用户态
  -> syscall wrapper
  -> ecall
  -> Trap
  -> 内核 syscall handler
  -> 返回用户态
```

`write`、`read`、`sleep`、`yield`、`exit` 都是这种模式。

这让我想到后端里的接口设计：用户不能直接操作数据库内部结构，只能通过 API 请求服务。系统调用就是操作系统层面的 API，只不过它跨越的是用户态和内核态。

## 7. trace 练习应该怎么讲

`sys_trace` 不是为了做一个很实用的系统调用，而是训练我们理解系统调用路径。

它要求实现三件事：

```text
读当前任务某个地址的 1 字节
写当前任务某个地址的 1 字节
查询当前任务调用某个 syscall 的次数
```

真正关键的是第三个：系统调用计数。

为什么计数要放在 TCB 里？

因为每个任务应该有自己的统计。如果放全局变量里，app0 和 app1 的 syscall 次数会混在一起。

为什么计数要在 `handle_syscall()` 一开始做？

因为所有 syscall 都会经过这里，而且题目要求 trace 本次调用也要算进去。

因此：

```text
handle_syscall:
  先根据 a7 拿 syscall id
  对当前任务的 syscall_count[id] += 1
  再分发到具体 syscall 实现
```

这道题让我明白：很多内核功能不是在业务函数里硬写，而是在统一路径上加逻辑。系统调用统计就是挂在统一入口 `handle_syscall()` 上。

## 8. 为什么 trace 读写是不安全的

ch3 里 `trace_read` 和 `trace_write` 会把用户给的地址直接当指针用。比如：

```text
id -> *const u8
id -> *mut u8
```

这在真实系统里很危险。因为用户可能传一个非法地址，内核直接解引用就会出错。

为什么本章还能这样做？

因为 ch3 还没有地址空间隔离，课程目标不是马上做安全检查，而是先理解系统调用和任务结构。到了 ch4，会引入页表、虚拟地址和权限检查，才能更正规地判断用户地址是否合法。

所以这也是章节之间的逻辑：

```text
ch3:
  先让任务和 syscall 跑起来。

ch4:
  再解决地址安全和隔离。
```

## 9. 贪吃蛇为什么是一个好实验

贪吃蛇看起来是游戏，但它刚好需要操作系统的几个基本服务：

```text
输入:
  玩家按 W/A/S/D/Q。

输出:
  画蛇、食物、边框和分数。

计时:
  每隔一段时间移动一格。

调度:
  用户程序不能独占 CPU。
```

所以它不是随便写小游戏，而是在验证：

```text
用户态程序能否通过 syscall 使用内核服务？
内核能否给用户程序提供输入输出设备抽象？
分时调度是否还能正常工作？
```

## 10. 图形链路怎么讲给别人

我会这样讲图形链路：

用户态不直接知道 GPU 怎么工作。用户态只知道自己要提交一帧游戏状态。

这一帧用结构体表示：

```text
SnakeFrame:
  magic
  width / height
  snake length
  score
  food position
  snake body positions
```

然后用户态调用：

```text
write(fd=3, SnakeFrame)
```

内核发现 `fd=3`，就知道这不是普通文本输出，而是图形帧，于是：

```text
检查 magic
解析 SnakeFrame
画背景
画边框
画食物
画蛇
画分数
flush 到 framebuffer
```

最后 QEMU 窗口显示出来。

这条链路体现的是“设备抽象”：用户态不碰设备寄存器，内核负责把用户请求转换为设备操作。

## 11. 键盘链路怎么讲给别人

键盘输入也是类似思想。

用户态并不直接读键盘设备。QEMU 产生键盘事件，内核轮询 VirtIO Keyboard，把 keycode 转成普通字符：

```text
W -> b'w'
A -> b'a'
S -> b's'
D -> b'd'
Q -> b'q'
```

然后用户程序通过 `read(STDIN)` 读到这个字符。

完整链路：

```text
QEMU 窗口按键
  -> VirtIO Keyboard
  -> keyboard.rs
  -> input cache
  -> read(STDIN)
  -> try_getchar()
  -> snake 改方向
```

这就和真正系统里的思想类似：应用看到的是标准输入，内核背后可以接 UART、键盘、文件、管道等不同来源。

## 12. 为什么本地和 CI 要分开

本地我要看到图形窗口，所以 `.cargo/config.toml` 里用：

```text
-display gtk
-device virtio-gpu-device
-device virtio-keyboard-device
```

但是 GitHub Actions 没有图形桌面。如果 CI 也用 GTK，就可能直接失败或卡住。

所以 `test.sh` 里覆盖 runner，用：

```text
-display none
-serial stdio
```

这样 CI 不开窗口，只看串口输出是否包含：

```text
Test ch3 snake OK!
```

这就是工程上常见的“本地体验”和“自动测试”分离。

## 13. 我如何用 AI 协作完成本章

这次不是单纯让 AI 直接写完代码，而是采用了逐步问答的方式：

```text
先问我概念:
  yield 和 exit 有什么区别？
  为什么需要时钟中断？
  TCB 为什么要保存 syscall_count？

再让我解释代码链:
  用户态 write 怎么进入内核？
  fd=3 怎么变成图形输出？
  read(STDIN) 怎么拿到键盘输入？

遇到 bug 再定位:
  QEMU 没图形是设备问题还是地址冲突？
  为什么 CI 不能跑交互版？
  为什么 Windows 本地找不到 qemu？
```

这种方式让我不是只拿到代码，而是知道每个改动背后的原因。

## 14. 如果我要讲给别人，推荐顺序

我会按这个顺序讲：

1. ch2 只是批处理，ch3 才开始多任务。
2. 多任务靠保存和恢复上下文实现。
3. TCB 是每个任务的内核档案。
4. yield 是主动让出 CPU。
5. timer interrupt 是内核强制切换。
6. syscall 是用户态请求内核服务的桥。
7. trace 练习是在 syscall 路径上加统计。
8. snake 是用户态程序，不应该塞进内核。
9. 图形输出通过 `write(fd=3)` 进入内核。
10. 键盘输入通过 VirtIO Keyboard 和 `read(STDIN)` 回到用户态。
11. `snake-ci` 是为了自动测试，交互版是为了人玩。

## 15. 一句话讲完 ch3

ch3 把系统从“顺序执行多个程序”推进到“分时管理多个任务”：内核为每个用户程序维护 TCB，保存和恢复上下文，通过 syscall 与时钟中断控制任务切换；在此基础上，用户态贪吃蛇通过内核提供的输入输出设备抽象完成了一个可交互 demo。
