# rCore ch3 学习复盘与问答底稿

本文记录我和 AI 协作学习 ch3 时反复确认的问题。它不是标准答案摘抄，而是按“我容易混乱的点”整理出来的复盘。写这份文档的目的不是为了显得我一开始就懂，而是记录我从模糊理解到能解释清楚的过程。

## 1. 问：ch3 相比 ch2 最大的变化是什么？

我的原始回答：

> ch2 是一个一个运行，ch3 是多个程序一起运行。

修正后的回答：

方向对，但需要更精确。ch3 不是让单核 CPU 真正同时执行多个程序，而是让多个程序同时驻留在内存中，内核通过保存和恢复上下文，在它们之间快速切换。

更准确地说：

```text
ch2:
  程序必须运行结束后，内核才加载下一个程序。

ch3:
  程序可以没结束就被暂停，之后还能恢复继续执行。
```

所以 ch3 的核心不是“多个程序存在”，而是“多个程序可以被调度管理”。

## 2. 问：为什么 ch3 要把应用放到不同地址？

我的原始回答：

> 因为一个地址只能放一个程序，多个程序放一起会覆盖。

修正后的回答：

是的，但还要补一句：ch3 中 app0 暂停后以后还要继续运行，所以 app1 不能覆盖 app0 的代码和数据。

ch2 可以复用同一地址，是因为前一个程序已经结束。ch3 不能这样，因为多个程序要同时驻留内存。

```text
app0 -> 0x80400000
app1 -> 0x80600000
app2 -> 0x80800000
```

这一章还没有页表，所以这里只是物理地址层面的简单隔开，不是真正安全隔离。

## 3. 问：TCB 到底是什么？

我的原始困惑：

> 上下文、任务、栈、TCB 这些名词感觉来回绕。

修正后的回答：

TCB 是 Task Control Block，任务控制块。它是内核为每个任务准备的管理结构，不是用户程序本身。

可以这样拆：

```text
用户程序:
  要执行的代码和数据。

任务:
  用户程序的一次执行实体。

TCB:
  内核为了管理这个执行实体而建立的档案。

上下文:
  TCB 里保存的寄存器现场。
```

我可以用一个类比记住：

```text
用户程序像学生本人。
TCB 像学校系统里这个学生的档案。
上下文像这个学生当前正在做哪道题、草稿纸写到哪里。
```

## 4. 问：TCB 里为什么需要 stack？

我的原始理解：

> 每个程序要有自己的栈，不然函数调用会乱。

修正后的回答：

对。每个任务都有自己的函数调用、局部变量、返回地址。如果多个任务共用一块栈，任务切换后栈内容会互相覆盖。

所以每个 TCB 里要有独立用户栈：

```text
task0.stack
task1.stack
task2.stack
```

这样恢复 task0 时，它还能看到自己原来的函数调用链和局部变量状态。

## 5. 问：上下文保存到底保存了什么？

我的原始回答：

> 保存寄存器，之后切回来继续跑。

修正后的回答：

这句话对，但可以更具体。上下文至少要包含：

```text
程序执行位置:
  sepc / pc

栈位置:
  sp

函数调用返回位置:
  ra

参数和返回值:
  a0-a7

其他计算状态:
  通用寄存器
```

这些信息共同决定“程序下次从哪里继续、继续时变量和调用状态是什么”。

## 6. 问：yield 和 exit 有什么区别？

我的原始回答：

> yield 是让出 CPU，exit 是任务结束。

修正后的回答：

正确。可以进一步说：

```text
yield:
  当前任务仍然是可运行状态。
  内核之后还会调度它。

exit:
  当前任务已经完成。
  内核把它标记为 finish，不再调度。
```

这两个 syscall 都会让控制权回到内核，但内核对任务的处理方式不同。

## 7. 问：为什么只靠 yield 不够？

我的原始回答：

> 如果程序不主动 yield，就会一直占着 CPU。

修正后的回答：

对。只靠 yield 是协作式调度，它依赖用户程序主动让出 CPU。如果程序恶意或写错，系统就会卡住。

因此需要时钟中断：

```text
用户程序正在运行
  -> 时间片到
  -> CPU 自动 Trap 到内核
  -> 内核切换任务
```

这就是抢占式调度。它让操作系统有能力强制拿回 CPU。

## 8. 问：系统调用为什么不是普通函数调用？

我的原始理解：

> 用户态要请求内核，所以要 ecall。

修正后的回答：

对。普通函数调用仍然在同一个特权级里，而系统调用要从用户态进入内核态。

用户程序不能直接调用内核函数，因为这会破坏隔离和安全。它只能通过受控入口 `ecall` 进入内核。

```text
用户态函数
  -> syscall wrapper
  -> ecall
  -> Trap 到内核
  -> 内核 syscall 分发
```

所以 syscall 是“跨特权级的函数请求”，不是普通函数调用。

## 9. 问：sys_trace 为什么要记录每个任务自己的 syscall 次数？

我的原始回答：

> 因为题目问当前任务，所以不能全局统计。

修正后的回答：

正确。`sys_trace` 查询的是当前任务调用某个 syscall 的次数。如果用全局数组：

```text
global_syscall_count[id]
```

那 app0、app1、app2 的调用次数会混在一起。

所以要放在 TCB 里：

```rust
syscall_count: [usize; 512]
```

每个任务一份，互不影响。

## 10. 问：trace_request = 2 为什么本次 trace 也要计入？

我的原始困惑：

> 查询次数的时候，本次查询也算吗？

修正后的回答：

题目明确要求“本次调用也计入统计”。所以计数必须发生在分发到 `Trace::trace` 之前。

正确顺序：

```text
用户调用 trace
  -> ecall
  -> handle_syscall 先 syscall_count[410] += 1
  -> Trace::trace 查询 syscall_count[410]
```

如果先查再加，测试会少 1。

## 11. 问：trace_read / trace_write 为什么不安全？

我的原始回答：

> 因为现在没有地址空间隔离。

修正后的回答：

正确。ch3 还没有页表，内核不知道用户传来的地址是否合法。用户如果传一个错误地址，内核直接解引用就可能崩溃。

本章暂时允许这样做，是因为教学重点是 syscall 和任务管理。ch4 会引入：

```text
虚拟地址
物理地址
页表
地址空间
权限位
```

到那时才能更安全地检查用户地址。

## 12. 问：为什么贪吃蛇要做成用户态程序？

我的原始回答：

> 游戏是应用程序，不应该塞进内核。

修正后的回答：

对。课程要求是“实现用户态游戏应用，并扩展内核支持它”。这意味着：

```text
游戏规则:
  放在用户态。

输入输出机制:
  由内核提供。
```

如果把蛇的移动、食物、碰撞都写进内核，那只是“内核画了一个动画”，没有体现用户程序使用操作系统服务。

## 13. 问：snake 用户态程序负责什么？

答案：

用户态 `ch3_snake.rs` 负责策略和游戏逻辑：

```text
蛇身数组
蛇头方向
食物位置
分数
按键处理
碰撞检测
每帧打包 SnakeFrame
```

它不需要知道 GPU 设备寄存器，也不需要知道 VirtIO 队列细节。它只需要调用系统调用。

## 14. 问：snake 内核态负责什么？

答案：

内核负责机制和设备抽象：

```text
graphics.rs:
  初始化 VirtIO-GPU
  建立 framebuffer
  接收 SnakeFrame
  绘制画面
  flush 到 QEMU 窗口

keyboard.rs:
  初始化 VirtIO Keyboard
  读取按键事件
  转成 w/a/s/d/q

main.rs:
  在 write(fd=3) 时转入 graphics
  在 read(STDIN) 时返回键盘缓存
```

这就是“内核提供机制，应用实现策略”。

## 15. 问：为什么使用 write(fd=3) 作为图形输出？

我的原始困惑：

> 为什么不直接让用户程序调用 graphics 函数？

修正后的回答：

因为用户程序不能直接调用内核函数，也不能直接访问内核模块。它只能通过 syscall。

所以我们约定：

```text
fd = 1: 普通 stdout
fd = 2: debug/stderr
fd = 3: 图形帧输出通道
```

用户态调用：

```text
write(3, SnakeFrame)
```

内核在 `SyscallContext::write()` 里识别 `fd=3`，把数据交给图形模块。这是一个教学版的“设备文件”思想。

## 16. 问：SnakeFrame 为什么要有 magic？

答案：

`magic` 可以理解为数据包头，用来确认用户传来的确实是 snake 图形帧。

如果没有 magic，内核只能看到一段字节，不知道它是不是合法的 `SnakeFrame`。加入 magic 后，内核可以做最基本的校验：

```text
magic 正确 -> 按 SnakeFrame 解析
magic 错误 -> 返回 -1
```

这虽然不是完整安全检查，但比完全相信用户数据更清楚。

## 17. 问：为什么一开始 QEMU 开了但没图？

我的原始猜测：

> 是不是路径问题？是不是环境变量问题？

修正后的分析：

路径和环境变量确实可能导致 QEMU 启动失败，但这次更关键的问题是 ch3 没有地址空间隔离，加入 GPU 后内核静态内存和 DMA 使用变多，用户程序如果仍放在较低地址，可能覆盖或冲突。

解决方式是把 snake case 的 app base 调高：

```toml
base = 0x8100_0000
```

这件事也提醒我：没有虚拟内存时，内存布局非常脆弱。很多“玄学 bug”本质上是地址冲突。

## 18. 问：为什么按键从 UART 改成 VirtIO Keyboard？

我的原始想法：

> 终端里能输入 WASD，应该也可以。

修正后的回答：

可以，但体验不好。终端输入和输出混在一起，游戏画面刷新会有残影。图形窗口里玩游戏更符合 demo 目标。

所以改成：

```text
QEMU GTK 窗口
  -> virtio-keyboard-device
  -> keyboard.rs
  -> read(STDIN)
```

这样用户点击 QEMU 窗口后，W/A/S/D/Q 就是游戏输入。

## 19. 问：为什么要有 snake-ci？

我的原始理解：

> CI 不能等人按键，所以要自动退出。

修正后的回答：

完全正确。交互版为了人玩，应该一直等输入；CI 版为了机器测，必须自动结束。

所以：

```text
snake:
  本地交互，QEMU GTK 窗口，W/A/S/D/Q 控制。

snake-ci:
  自动跑固定帧数，输出 Test ch3 snake OK! 后退出。
```

这个拆分保证了本地体验和自动测试都能成立。

## 20. 问：为什么本地 cargo run 和 CI runner 不一样？

答案：

本地需要图形窗口：

```text
-display gtk
```

CI 没有桌面环境，需要 headless：

```text
-display none
```

所以 `.cargo/config.toml` 面向本地体验，`test.sh` 在 CI 中覆盖 runner。这样不会因为本地显示配置导致 GitHub Actions 卡住。

## 23. 最终自检问答

问题：ch3 的主线是什么？

答案：让多个用户程序同时驻留内存，并通过 TCB、系统调用和时钟中断实现分时调度。

问题：为什么需要保存上下文？

答案：因为任务暂停后以后还要继续运行，必须保存寄存器、栈指针和执行位置。

问题：为什么系统调用计数放在 TCB？

答案：因为统计对象是当前任务，每个任务应有独立计数。

问题：为什么 trace 读写不安全？

答案：因为 ch3 没有页表，内核无法判断用户地址是否合法。

问题：snake 的游戏逻辑在哪里？

答案：在用户态 `ch3_snake.rs`。

问题：snake 的图形输出在哪里实现？

答案：用户态通过 `write(fd=3)` 提交帧，内核 `graphics.rs` 画到 VirtIO-GPU framebuffer。

问题：snake 的键盘输入从哪里来？

答案：QEMU VirtIO Keyboard 产生事件，内核 `keyboard.rs` 读取后通过 `read(STDIN)` 交给用户态。

问题：为什么要把 snake app 放到 `0x81000000`？

答案：为了避免 ch3 无地址空间隔离时，用户程序与内核图形/DMA 数据区域冲突。

问题：为什么要有 `snake-ci`？

答案：交互版会等待用户输入，不适合 CI；自动版能跑完并输出固定标志。
