# rCore ch6 补充讲述稿：从“能运行进程”到“能保存世界”

## 开场

如果说 ch5 让操作系统像 Linux，是因为它有了进程；那么 ch6 让操作系统更像真正能用的系统，是因为它有了文件。

没有文件系统时，程序只能活在内存里。程序一退出，数据就消失。进程之间也很难用一种统一方式交换长期数据。

有了文件系统之后，程序可以说：

```text
我想打开这个名字的文件。
我想读它。
我想写它。
我想关闭它。
我想创建另一个名字指向同一份内容。
我想删除一个名字。
我想查看这个文件的信息。
```

这就是 ch6 的任务。

## 文件不是“磁盘文件”这么简单

我一开始容易把文件理解成 Windows 里的 `.txt`、`.docx`。但操作系统里的文件抽象要更大。

在 Unix 思想里，文件是一个统一的 I/O 入口。你可以对它 read，也可以对它 write。

所以：

```text
控制台可以像文件。
磁盘文件可以像文件。
管道可以像文件。
设备也可以像文件。
```

这就是为什么本实验里我们能把 breakout 的图形输出做成 `fd=3`。用户程序不需要直接知道 VirtIO-GPU 怎么初始化，它只要往 `fd=3` 写一帧结构体，内核就知道这是图形帧。

## fd 是一张“当前进程自己的表”

文件描述符 fd 不是文件本体。

它更像：

```text
当前进程打开文件表里的编号。
```

比如一个进程里：

```text
fd_table[0] = stdin
fd_table[1] = stdout
fd_table[3] = file "a.txt"
fd_table[4] = file "b.txt"
```

另一个进程里，`fd=3` 可能指向完全不同的文件。

这和 PID 类似：PID 是内核识别进程的编号；fd 是进程内部识别已打开 I/O 对象的编号。

## UserBuffer 的直觉

用户程序调用：

```rust
write(fd, buf, len)
```

这里的 `buf` 是用户虚拟地址。内核如果直接把它当指针用，就会犯 ch4 已经提醒过的错误：地址空间不同。

所以内核必须先问当前进程的页表：

```text
这个用户虚拟地址，在当前进程里对应哪个真实物理页？
这个页有没有读权限或写权限？
```

只有翻译成功后，内核才能安全地读写用户传来的数据。

这就是 UserBuffer/地址翻译存在的原因。

## 从 write 看完整链路

一次普通文件写入可以讲成这样：

```text
用户程序：我要写 fd=3
内核：先看当前进程是谁
内核：查它的 fd_table[3]
内核：发现这是一个 OSInode
内核：再翻译用户传来的 buf
内核：把字节交给 inode.write_at
easy-fs：找到文件对应的数据块
block cache：把块拿到内存里修改
virtio-blk：最终同步到 fs.img
```

这条链路非常重要，因为它把 ch4 的地址空间、ch5 的进程、ch6 的文件系统串起来了。

## easy-fs 的核心结构

easy-fs 可以理解成一套“如何在块设备上组织文件”的规则。

块设备只认识块：

```text
block 0
block 1
block 2
...
```

但人类和程序想看到的是：

```text
/filea
/fileb
/breakout.sav
```

文件系统的工作就是把“名字”和“块”联系起来。

关系大概是：

```text
文件名
-> 目录项 DirEntry
-> inode 编号
-> DiskInode
-> 数据块编号
-> 块设备内容
```

## inode 不等于文件名

这是 ch6 很容易绕晕的地方。

文件名只是目录里的一个名字。真正记录文件大小、类型、数据块位置的是 inode。

硬链接就是利用这一点：

```text
"a.txt" -> inode 5
"b.txt" -> inode 5
```

两个名字指向同一个 inode，所以它们看到的是同一份文件内容。

删除一个名字时，如果还有另一个名字指向这个 inode，就不能把数据块释放掉。

所以需要 `nlink`。

## linkat 和 unlinkat

`linkat` 做的是增加一个目录项。

它不是复制文件内容。

```text
linkat("a", "b")
-> 找到 a 的 inode
-> 创建目录项 b
-> b 指向 a 的 inode
-> inode.nlink += 1
```

`unlinkat` 做的是删除一个目录项。

```text
unlinkat("b")
-> 删除目录项 b
-> inode.nlink -= 1
-> 如果 nlink == 0，才释放 inode 和数据块
```

这就是为什么 ch6 练习不仅改内核系统调用，还改了 easy-fs 的底层结构。

## fstat

`fstat` 是通过 fd 查询文件元信息。

它回答的不是“文件内容是什么”，而是：

```text
这个 fd 指向哪个 inode？
它是文件还是目录？
它有几个硬链接？
```

内核拿到这些信息后，要写回用户传来的 `Stat` 结构体指针。这个指针仍然是用户地址，所以又需要地址翻译。

## breakout 为什么适合 ch6

打砖块游戏的核心逻辑并不复杂：

```text
挡板左右移动
小球反弹
砖块消失
分数增加
生命减少
```

真正和 ch6 相关的是保存和恢复：

```text
按 S
-> open("breakout.sav", CREATE | WRONLY)
-> write SaveData
-> close

按 R
-> open("breakout.sav", RDONLY)
-> read SaveData
-> 恢复游戏状态
-> close
```

这比单纯打印“文件系统通过”更直观：我能看到游戏状态被写进文件系统，再从文件系统读回来。

## 本章一句话总结

ch6 不是只多了几个 syscall。它把用户程序、进程 fd 表、虚拟地址翻译、文件对象、inode、block cache、virtio-blk 和 fs.img 全部连成了一条数据通路。

真正理解这一章，就是能讲清楚这句话：

```text
用户程序写一个 fd，内核如何一路找到真实块设备，并把数据保存下来。
```
