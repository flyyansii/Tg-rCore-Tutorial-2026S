# rCore ch8 学习复盘与问答底稿

## 本章主题

第八章的核心是线程与同步。到 ch7 为止，系统已经有进程、地址空间、文件系统、管道、重定向、信号等能力，但“一个进程内部同时执行多条控制流”的能力还不完整。ch8 把原来“进程既是资源容器又是执行单元”的模型拆开：

```text
Process：资源容器，管理地址空间、文件描述符、信号、同步原语列表。
Thread：执行单元，管理 TID、寄存器上下文、satp 和调度状态。
```

这章的理解主线是：

```text
进程负责“有什么资源”
线程负责“谁在执行”
同步原语负责“多个线程如何安全协作”
死锁检测负责“发现资源等待环路并拒绝危险申请”
```

## 和前几章的关系

ch5 讲进程，重点是 `fork/exec/waitpid`，一个进程对应一个主要执行流。

ch6 讲文件系统，重点是 fd、inode、块设备和 easy-fs。

ch7 讲管道、命令行参数和重定向，重点是进程间通信和 shell 组合能力。

ch8 在这些基础上引入线程：

```text
ch5：多个进程并发。
ch6：进程通过文件系统访问持久化数据。
ch7：进程之间通过 pipe 通信。
ch8：同一进程内部多个线程共享资源并发执行。
```

## 问答复盘

### 问题 1：为什么要把 Process 和 Thread 分开？

我的初始理解是：线程像是进程里面更小的执行单位。

修正后的答案：

这个理解方向对，但还需要更精确。进程和线程的分工是：

- `Process` 保存共享资源：地址空间、文件描述符表、信号处理器、mutex/semaphore/condvar 列表。
- `Thread` 保存执行状态：线程 ID、上下文、用户栈、当前执行到哪里。

同一进程内的多个线程共享同一个 `Process`，所以它们共享地址空间和 fd 表；但每个线程都有独立 `Thread`，所以它们能在不同函数位置、不同用户栈上并发运行。

### 问题 2：为什么线程要有自己的用户栈？

我的初始理解是：不同线程如果共用一个栈，函数调用和局部变量会混在一起。

修正后的答案：

完全正确。线程共享代码段、数据段、堆，但不能共享运行栈。因为每个线程都有自己的函数调用链、返回地址、局部变量和临时数据。ch8 的 `thread_create` 会在当前进程地址空间的高地址区域向下搜索空闲页，为新线程映射独立用户栈。

### 问题 3：PThreadManager 是什么？

我的初始理解是：它是一个同时管理进程和线程的调度器。

修正后的答案：

`PThreadManager` 是 ch8 的全局管理核心。它不是单纯的调度队列，而是“双层管理器”：

```text
ProcManager：PID -> Process
ThreadManager：TID -> Thread，并维护 ready_queue
PThreadManager：维护 PID/TID 之间的归属关系，负责 current、阻塞、唤醒、退出、等待。
```

也就是说，调度粒度从“进程”变成“线程”，但资源归属仍然要回到进程。

### 问题 4：thread_create 做了什么？

我的初始理解是：创建一个新线程，然后让它从某个函数入口开始执行。

修正后的答案：

`thread_create(entry, arg)` 的流程是：

```text
找到当前进程
在当前进程地址空间里找空闲用户栈区域
映射新线程用户栈
构造 LocalContext::user(entry)
设置 sp 为新用户栈顶
设置 a0 = arg
用相同 satp 构造 Thread
把 Thread 加入 PThreadManager
返回 tid
```

这里最关键的是：新线程和当前线程使用同一个地址空间 `satp`，因此它们共享进程内存。

### 问题 5：mutex 和 semaphore 有什么区别？

我的初始理解是：mutex 是只能一个线程进入，semaphore 是资源数量可以大于 1。

修正后的答案：

这很接近准确答案。`Mutex` 是互斥锁，通常表示一个临界区只能有一个线程持有。`Semaphore` 是计数型资源，初始值可以是 N，表示同一类资源有 N 份。`semaphore_down` 类似申请资源，`semaphore_up` 类似归还资源。

代码里：

- `mutex_list` 存放当前进程创建的互斥锁。
- `semaphore_list` 存放当前进程创建的信号量。
- 它们都属于 `Process`，因此同一进程内所有线程共享。

### 问题 6：阻塞锁为什么会让线程退出就绪队列？

我的初始理解是：如果资源拿不到，线程继续运行也没意义，所以让它等。

修正后的答案：

对。ch8 主循环中，如果 `SEMAPHORE_DOWN`、`MUTEX_LOCK` 或 `CONDVAR_WAIT` 返回 `-1`，表示当前资源不可用，内核会调用 `make_current_blocked()`，把当前线程从可运行状态切走。等资源释放时，`semaphore_up` 或 `mutex_unlock` 会返回一个需要唤醒的 TID，内核再调用 `re_enque(tid)` 把它放回 ready queue。

### 问题 7：condvar_wait 为什么要和 mutex 配合？

我的初始理解是：条件变量不是直接保护资源，而是等待某个条件。

修正后的答案：

条件变量用于“等待某个条件成立”，但条件本身通常由共享数据决定。访问共享数据必须用 mutex 保护。所以 `condvar_wait(condvar_id, mutex_id)` 的典型语义是：

```text
当前线程持有 mutex
发现条件不满足
释放 mutex
阻塞到 condvar 等待队列
被 signal 唤醒
重新竞争 mutex
继续检查条件
```

这就是为什么条件变量不能脱离互斥锁单独理解。

### 问题 8：死锁是什么？

我的初始理解是：多个线程互相等资源，谁也走不下去。

修正后的答案：

死锁可以理解为资源等待图中出现环。例如：

```text
线程 A 持有锁 1，等待锁 2。
线程 B 持有锁 2，等待锁 1。
```

此时 A 等 B，B 等 A，两个线程都无法继续。ch8 的 exercise 要求实现 `enable_deadlock_detect`，开启后，如果 `mutex_lock` 或 `semaphore_down` 检测到可能死锁，就拒绝本次资源申请并返回 `-0xDEAD`。

### 问题 9：为什么死锁检测可以用 Available / Allocation / Need？

我的初始理解是：它像银行家算法，判断资源分配后系统是否安全。

修正后的答案：

是的。题目给出的算法就是一种安全性检测思路：

- `Available` 表示每类资源当前还剩多少。
- `Allocation` 表示每个线程已经持有哪些资源。
- `Need` 表示每个线程还需要哪些资源才能继续。

如果能找到一个顺序，让所有线程最终都能获得需要的资源、运行结束并释放资源，则系统安全；否则说明存在死锁或不安全状态。

### 问题 10：为什么 ch8 exercise 要允许开关死锁检测？

我的初始理解是：为了兼容和灵活，只有测试时打开。

修正后的答案：

对。死锁检测有额外开销，不一定所有程序都需要。`enable_deadlock_detect(is_enable)` 允许当前进程按需开启或关闭检测。开启后才对 mutex 和 semaphore 的资源申请进行安全性检查。

## 本章需要记住的十个点

1. ch8 的核心是从进程模型扩展到线程模型。
2. `Process` 管资源，`Thread` 管执行。
3. 同一进程的线程共享地址空间和 fd 表。
4. 每个线程必须有独立用户栈。
5. 调度粒度变成线程，而不是进程。
6. mutex 用于互斥，semaphore 用于计数资源。
7. condvar 用于等待条件，通常必须配合 mutex。
8. 阻塞线程会从 ready queue 中移除。
9. 解锁或释放资源时可能唤醒等待线程。
10. 死锁检测本质上是在资源分配前判断系统是否会进入不安全状态。
