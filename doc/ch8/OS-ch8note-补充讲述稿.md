# OS ch8 note 补充讲述稿

## 1. 这一章到底在解决什么问题

到 ch7 为止，我们已经能让多个进程运行，也能让进程通过文件、管道、重定向互相配合。但这里还有一个问题：如果一个程序内部希望同时做多件事，难道必须 fork 多个进程吗？

进程之间虽然隔离性强，但创建和切换成本更高，而且共享数据不方便。线程就是为了解决这个问题出现的：

```text
多个线程属于同一个进程。
它们共享地址空间和文件描述符。
但每个线程有自己的执行位置和用户栈。
```

所以 ch8 的主题是：在已有进程系统上，引入更细粒度的执行单位，并提供同步工具保证多个线程安全协作。

## 2. Process 和 Thread 的分离

前面章节中，我容易把“进程”理解成一个正在运行的程序。这个说法在前几章没太大问题，但 ch8 开始要更细：

```text
进程不是执行本身，而是资源容器。
线程才是真正被调度执行的单位。
```

代码中：

```text
Process:
  pid
  address_space
  fd_table
  signal
  semaphore_list
  mutex_list
  condvar_list

Thread:
  tid
  ForeignContext
```

这就像一个项目组：

```text
Process 是项目组的共享资料室。
Thread 是项目组里具体干活的人。
```

多人可以共享同一间资料室，但每个人手头正在做的事情不同。

## 3. 线程为什么共享地址空间

同一个进程内的线程共享地址空间，所以它们可以直接访问同一份全局变量、堆数据和文件描述符。这是线程的方便之处，也是危险之处。

方便在于：

```text
线程之间传数据不用 pipe。
一个线程改了堆里的数据，另一个线程马上能看到。
```

危险在于：

```text
多个线程同时改同一个变量，可能出现竞态。
```

比如两个线程都对 `counter += 1`，如果没有锁保护，可能两个线程都读到旧值，然后都写回相同的新值，导致计数丢失。

## 4. 线程为什么不能共享栈

线程共享地址空间，但不能共享栈。原因是栈保存的是每个线程自己的函数调用现场：

- 当前函数的局部变量。
- 返回地址。
- 临时保存的寄存器。
- 调用链。

如果多个线程共用一个栈，就像几个人同时在同一张草稿纸上写不同题目，很快就互相覆盖。

因此 `thread_create` 必须为新线程分配独立用户栈。

## 5. thread_create 的直观理解

`thread_create(entry, arg)` 可以理解成：

```text
请内核在当前进程里新开一条执行流，
让它从 entry 这个函数开始执行，
并把 arg 当作第一个参数传进去。
```

内核需要做：

```text
找当前进程
找一块还没用过的用户栈空间
映射这块栈
构造用户态上下文
设置入口 PC = entry
设置 sp = 新栈顶
设置 a0 = arg
创建 Thread
加入调度队列
```

这和创建进程不同。创建线程不需要复制整个地址空间，因为它本来就和当前线程共享同一个进程。

## 6. 为什么需要 mutex

mutex 是互斥锁，用来保护临界区。

临界区就是：

```text
同一时间只能有一个线程进入的代码区域。
```

比如多个线程一起修改全局变量：

```text
lock
counter += 1
unlock
```

如果没有锁，多个线程可能同时读写，结果不稳定。加锁后，一个线程修改时，其他线程必须等待。

## 7. 为什么需要 semaphore

semaphore 是信号量，适合表示“有多少份资源”。

mutex 通常只有 0/1 两种状态，而 semaphore 可以是任意计数：

```text
初始值为 3：表示资源有 3 份。
down：申请一份资源。
up：归还一份资源。
```

如果资源还有剩余，`down` 成功；如果资源已经用完，线程就需要阻塞等待。

## 8. 为什么需要 condvar

条件变量不是保护资源本身，而是让线程等待某个条件。

典型场景：

```text
消费者线程发现队列为空。
它不应该一直空转浪费 CPU。
它应该睡眠等待。
生产者放入数据后 signal 唤醒消费者。
```

条件变量通常和 mutex 配合：

```text
先用 mutex 保护共享队列。
发现条件不满足，就 condvar_wait。
wait 会释放 mutex 并阻塞当前线程。
signal 会唤醒等待线程。
```

## 9. 阻塞和唤醒

ch8 的同步原语不是简单返回成功/失败。资源不可用时，线程要进入阻塞态。

主循环中的关键逻辑是：

```text
mutex_lock / semaphore_down / condvar_wait 返回 -1
  -> 当前线程阻塞
  -> 从 ready queue 移除
  -> 调度其他线程

mutex_unlock / semaphore_up / condvar_signal 唤醒某个 tid
  -> re_enque(tid)
  -> 线程重新进入 ready queue
```

这就是同步原语和调度器之间的连接点。

## 10. 死锁为什么会出现

死锁的根源是资源等待形成环。

最简单例子：

```text
线程 A 拿了锁 1，想要锁 2。
线程 B 拿了锁 2，想要锁 1。
```

此时：

```text
A 等 B 释放锁 2。
B 等 A 释放锁 1。
```

两个线程都不会主动继续，于是系统卡住。

## 11. 死锁检测题目要求

ch8 exercise 要实现：

```text
enable_deadlock_detect(is_enable)
syscall ID = 469
```

当当前进程开启死锁检测后：

```text
mutex_lock
semaphore_down
```

在真正阻塞前要先判断：如果这次资源申请会导致死锁，就拒绝申请并返回：

```text
-0xDEAD
```

这样用户程序可以检测到危险情况，而不是永远卡死。

## 12. 银行家/安全性检测思路

题目给的检测方法可以这样理解：

```text
我先假装系统继续运行。
如果能找到一个执行顺序，让所有线程都拿到需要的资源并最终释放资源，
说明系统安全。
如果找不到，说明某些线程会互相等待，系统不安全。
```

核心数据：

```text
Available：当前可用资源数量。
Allocation：每个线程已经拿到多少资源。
Need：每个线程还需要多少资源。
Work：模拟运行时手头可用资源。
Finish：某个线程是否能顺利完成。
```

检测时不断找一个 `Need <= Work` 的线程，假装它能完成并释放资源。如果最后所有线程都能 Finish，说明安全；否则存在死锁风险。

## 13. ch8_usertest 的意义

`ch8_usertest` 是综合测试入口，它会 fork 多个子进程，然后 exec 各种测试：

- 基础程序测试。
- 线程创建测试。
- 带参数线程测试。
- 信号量同步测试。
- mutex 竞态测试。
- 哲学家就餐测试。
- 条件变量测试。
- 管道测试。
- 死锁检测测试。

它不是一个单独功能测试，而是检查 ch8 之前的能力和 ch8 新增能力能否一起工作。

## 14. 给别人讲这一章的顺序

我会这样讲：

1. 前几章的进程既像资源容器，又像执行单元。
2. ch8 把它拆成 Process 和 Thread。
3. Process 共享资源，Thread 独立执行。
4. 多线程共享内存带来竞态。
5. mutex、semaphore、condvar 用来协调线程。
6. 资源不可用时，线程进入阻塞态。
7. 资源释放时，等待线程被唤醒。
8. 如果资源等待形成环，就会死锁。
9. exercise 要实现 `enable_deadlock_detect`，在危险申请发生前拒绝。
10. 最终文档和实验过程整理后提交到 GitHub，作为 AI 协作学习记录。
