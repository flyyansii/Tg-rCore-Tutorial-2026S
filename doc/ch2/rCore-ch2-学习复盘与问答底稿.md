# rCore ch2 学习复盘与问答底稿

## 1. 学习主线

ch2 的学习主线是：在 ch1 内核能启动的基础上，让内核具备运行用户程序的能力。

我一开始容易把它理解成“把多个程序放到内核里，然后一个个执行”。这个说法不算错，但不够完整。更准确地说，ch2 建立了三个机制：

```text
构建期：把用户程序编译并链接进内核。
运行期：内核把用户程序复制到指定地址并进入 U-mode。
交互期：用户程序通过 ecall 请求内核服务。
```

这三个机制合在一起，就是批处理操作系统的最小闭环。

## 2. 问题：批处理系统和普通函数调用有什么区别？

我的初始回答：

> 批处理就是一个程序执行完再执行下一个程序。

补充后的理解：

批处理不只是顺序执行。关键区别是：用户程序和内核程序处在不同特权级。用户程序不能直接调用内核函数，只能通过 `ecall` 陷入内核。内核处理完请求后，再决定是回到当前用户程序，还是结束当前程序并运行下一个。

所以 ch2 的批处理系统不是：

```text
kernel 调 app1()
kernel 调 app2()
```

而是：

```text
kernel 准备 app1 的用户上下文
  -> sret 进入 U-mode
  -> app1 ecall 回内核
  -> app1 exit
  -> kernel 准备 app2 的用户上下文
```

## 3. 问题：用户程序为什么不能直接 `println!`？

我的初始回答：

> 因为用户程序没有标准库，需要系统调用。

补充后的理解：

用户态 `println!` 最终需要输出到终端，而终端是硬件/内核资源。用户程序没有权限直接访问 SBI console 或 UART，所以 `println!` 必须封装成 `write` 系统调用。

调用链是：

```text
println!
  -> user_lib console
  -> sys_write
  -> ecall
  -> 内核 SyscallContext::write
  -> SBI/console 输出
```

所以 `println!` 不是消失了，而是从“标准库帮我们做”变成“我们自己的 user_lib + 内核 syscall 帮我们做”。

## 4. 问题：`ecall` 是什么？

我的初始回答：

> ecall 是调用内核级执行指令。

补充后的理解：

`ecall` 是 RISC-V 提供的环境调用指令。用户程序执行 `ecall` 时，CPU 会从 U-mode 切换到 S-mode，并跳到内核设置好的 Trap 入口。

它不是普通函数调用，因为普通函数调用不会改变特权级；`ecall` 会触发 Trap，让 CPU 进入更高特权级处理。

```text
普通函数调用：jal -> 同一特权级
系统调用：ecall -> U-mode 到 S-mode
```

## 5. 问题：TrapContext 是干什么的？

我的初始理解：

> 保存用户程序执行到哪里，类似 ARM 里的 LR 或 PC 返回信息。

补充后的理解：

这个类比方向是对的，但 TrapContext 保存得更多。它保存的是用户态被打断时的完整现场，包括通用寄存器、`sstatus`、`sepc` 等。

可以这样记：

```text
sepc：用户程序被打断的位置
sstatus：特权状态，决定 sret 后回到哪个模式
x[0..31]：用户程序当时用到的寄存器
```

如果没有 TrapContext，内核处理完系统调用后就不知道用户程序该从哪里继续、寄存器该恢复成什么。

## 6. 问题：`sepc` 为什么要加 4？

我的初始理解：

> 为了跳过 ecall。

补充后的理解：

RISC-V 中普通指令通常是 4 字节。发生 `ecall` 时，`sepc` 保存的是 `ecall` 这条指令的地址。如果处理完系统调用后不把 `sepc += 4`，`sret` 回去后会重新执行同一条 `ecall`，于是又陷入内核，形成死循环。

所以：

```text
write/yield 这种要返回用户程序的 syscall：
  需要 sepc += 4

exit 这种不返回当前程序的 syscall：
  不需要回到原来的 ecall 后面
```

## 7. 问题：`scause`、`sepc`、`sstatus` 分别是什么？

### `scause`

记录 Trap 原因，例如：

```text
UserEnvCall：用户执行 ecall
IllegalInstruction：非法指令
StoreFault：非法写地址
```

### `sepc`

记录 Trap 发生时用户程序的 PC。

### `sstatus`

记录特权状态。`sret` 会参考它决定返回后处于 U-mode 还是 S-mode。

你之前用 ARM 的 CPSR/APSR/LR 来类比，这是有帮助的：

```text
sepc 类似“异常返回地址”
sstatus 类似“异常前后的状态寄存器信息”
scause 类似“异常原因编号”
```

## 8. 问题：用户程序是怎么被链接进内核的？

我的初始回答：

> build.rs 生成 link_app.S，把用户程序放进内核。

补充后的理解：

构建期流程是：

```text
读取 cases.toml
编译用户程序
得到 ELF/bin
生成 app.asm
app.asm 用 .incbin 包含用户程序字节
main.rs 用 global_asm!(APP_ASM) 把 app.asm 链进内核
```

运行期再通过 `AppMeta::locate()` 找到这些程序的起止地址。

所以这里有两个阶段：

```text
构建期：用户程序被放进内核镜像。
运行期：内核从镜像中找到用户程序并复制到运行地址。
```

## 9. 问题：为什么用户程序需要 `_start`，不能只写 `main`？

我的初始回答：

> 因为需要定义执行顺序，不然 main 返回后不知道去哪。

补充后的理解：

用户程序运行前也需要一个最小运行时。它负责：

```text
清空 .bss
调用 main
拿到 main 返回值
调用 exit
```

如果直接从 `main` 开始，`main` 返回后就没有统一收尾逻辑。`_start` 的存在让用户程序有明确生命周期：

```text
_start -> main -> exit
```

## 10. 问题：`sys_write` 为什么不直接写在用户程序 main 里？

我的初始困惑：

> syscall 为什么写 main 里，为什么不都放 syscall.rs？

补充后的理解：

用户态和内核态各有一套 syscall 相关代码。

用户态：

```text
user/src/syscall.rs
  负责把 syscall id 和参数放进寄存器，然后 ecall。
```

内核态：

```text
ch2/src/main.rs 或 syscall 模块
  负责读取 syscall id，分发到 write/exit 等实现。
```

两者不是重复，而是一问一答：

```text
用户态 syscall.rs：发请求
内核态 SyscallContext：处理请求
```

## 11. 问题：ch2 和 ch3 的区别是什么？

ch2：

```text
一次只运行一个 app。
当前 app exit 后才运行下一个。
没有时间片。
没有任务轮转。
```

ch3：

```text
多个 app 都初始化为任务。
每个任务有 TCB。
可以 yield 主动让出 CPU。
可以靠时钟中断强制切换。
```

所以 ch2 是“批处理”，ch3 是“分时多任务”。

## 12. ch2 学完以后应该能讲清楚什么

我应该能讲清楚：

1. 用户程序如何在构建期被打包进内核。
2. 内核如何找到用户程序的二进制边界。
3. 内核如何把用户程序复制到运行地址。
4. 用户程序如何从 `_start` 调到 `main`。
5. `println!` 如何变成 `sys_write`。
6. `ecall` 如何从用户态进入内核态。
7. TrapContext 为什么要保存用户现场。
8. `sepc += 4` 为什么必要。
9. `exit` 为什么会让内核运行下一个程序。
10. 组件化仓库中哪些 crate 对应 Guide 原文中的 trap/syscall/batch 模块。

## 13. 扩展问答：把 ch2 拆成 30 个自测问题

这一节是给后续复习用的。每个问题都尽量对应一个 Guide 里的关键点或我们对话中卡住过的点。

### 1. ch2 为什么不是“内核直接调用用户函数”？

因为用户程序运行在 U-mode，内核运行在 S-mode。内核不是用 Rust 函数调用进入用户程序，而是构造用户上下文后用 `sret` 降到用户态。这样才有特权级隔离。

### 2. 用户程序和内核为什么要分开编译？

它们的角色不同。内核是执行环境，用户程序是被执行对象。用户程序要依赖 `user_lib`，通过 syscall 请求服务；内核则要实现这些 syscall。

### 3. 为什么 ch2 还不能从磁盘加载应用？

因为 ch2 还没有文件系统，也没有块设备驱动。应用只能在构建期嵌入内核镜像，运行时由内核从镜像里取出来。

### 4. `cases.toml` 的意义是什么？

它告诉构建脚本本章应该打包哪些用户程序，相当于“批处理队列”的构建期配置。

### 5. `build.rs` 为什么这么重要？

它不是运行时内核代码，而是编译内核前执行的脚本。它负责把用户程序编译结果整理成内核可以链接的汇编数据。

### 6. `.incbin` 是什么？

`.incbin` 是汇编里的“包含二进制文件”指令。它把用户程序的二进制字节原封不动放进内核镜像。

### 7. `app_0_start/app_0_end` 保存的是什么？

它们保存 app0 在内核镜像中的字节范围，不是 app0 最终运行时 PC 的含义。

### 8. `base + i * step` 是什么？

这是内核把第 i 个应用复制到的运行地址。ch2 没有页表，所以这个地址基本按物理地址理解。

### 9. 为什么要复制 app，而不是直接在内核镜像中运行？

因为内核镜像中的 app 是作为数据被链接进来的，运行位置和链接预期不一定一致。批处理系统需要把 app 放到约定运行地址，保证入口和内部地址布局正确。

### 10. ch2 的用户程序有没有自己的 `_start`？

有。用户程序也需要入口，负责清 `.bss`、调 `main`、最后 `exit`。

### 11. `main` 返回后为什么不能直接结束？

裸机环境下没有操作系统替你回收。用户程序必须通过 `exit` 告诉内核“我结束了”，否则内核不知道什么时候运行下一个程序。

### 12. 用户态 `println!` 和内核态 `println!` 一样吗？

不一样。用户态 `println!` 走 `sys_write -> ecall`，内核态 `println!` 可以直接走内核 console/SBI。

### 13. `syscall.rs` 为什么在用户目录里？

用户目录里的 `syscall.rs` 是“发请求”的库。它只负责按 RISC-V ABI 放参数并执行 `ecall`，不负责真正写屏幕。

### 14. 内核里的 syscall 实现负责什么？

内核负责“处理请求”。例如 `write` 要把用户 buffer 中的数据打印到控制台，`exit` 要结束当前 app 并准备下一个 app。

### 15. `ecall` 为什么不是普通函数调用？

普通函数调用不会改变特权级；`ecall` 会让 CPU 进入 Trap 流程，从 U-mode 切到 S-mode。

### 16. `stvec` 起什么作用？

`stvec` 保存 Trap 入口地址。用户执行 `ecall` 后，CPU 会跳到 `stvec` 指向的内核入口。

### 17. `scause` 起什么作用？

`scause` 告诉内核 Trap 的原因。是系统调用、非法指令、访存错误还是时钟中断，内核要根据它分发处理。

### 18. `sepc` 起什么作用？

`sepc` 保存用户程序被打断时的 PC。处理完 Trap 后，`sret` 会根据它回到用户程序。

### 19. `sstatus` 起什么作用？

`sstatus` 保存特权状态和中断状态。`sret` 返回时要依赖它决定回到 U-mode 还是 S-mode。

### 20. TrapContext 为什么必须保存通用寄存器？

因为用户程序执行到一半时寄存器里有局部计算结果和参数。如果不保存，内核处理 syscall 时会覆盖它们，用户程序回来就乱了。

### 21. `a7` 为什么重要？

RISC-V syscall 约定用 `a7` 存 syscall id。内核通过它知道用户请求的是 `write`、`exit` 还是其他服务。

### 22. `a0` 为什么既是参数又是返回值？

这是调用约定。进入 syscall 时 `a0` 可作为第一个参数；返回时内核把返回值写回 `a0`。

### 23. 为什么 `write` 后要回用户态？

因为 `write` 只是服务请求，用户程序还没结束。内核打印完以后应当恢复用户程序继续执行。

### 24. 为什么 `exit` 后不回用户态？

因为 `exit` 表示当前用户程序生命周期结束。内核应该丢弃当前 app 的运行状态，转向下一个 app。

### 25. ch2 中异常程序会怎样？

如果用户程序非法访存或执行非法指令，Trap 原因不是普通 syscall。内核通常会杀死该应用，然后继续批处理后面的应用。

### 26. ch2 的“批处理”体现在哪里？

内核维护一组 app，并且按顺序运行。app0 结束后才运行 app1，app1 结束后才运行 app2。

### 27. ch2 为什么不能做到 app0 没结束就跑 app1？

因为它还没有保存多个任务现场和调度器。它只有“当前 app”概念，没有 TCB 数组。

### 28. 组件化仓库为什么看不到 Guide 那么多文件？

因为 `trap/syscall/linker/context` 被抽到 crate 里或集中到 `main.rs`。概念没消失，只是文件布局变了。

### 29. 我该如何对比 Guide 和组件化版本？

先按 Guide 理解模块职责，再在组件化仓库里找对应 crate 或 trait 实现。不要因为文件少就以为机制少。

### 30. ch2 最终要掌握的一句话是什么？

ch2 的本质是让内核拥有“加载用户程序、进入用户态、处理用户请求、结束后运行下一个程序”的能力。

## 14. 我之前理解中的容易偏差

1. 我容易把“app 链接进内核”理解成“app 变成内核函数”，这是不准确的。app 是被嵌入为字节数据，运行时仍然作为用户程序执行。
2. 我容易把 `ecall` 理解成“调用内核函数”，但它其实是特权级切换和 Trap，不是普通 call。
3. 我容易把 `TrapContext` 类比成一个返回地址，但它比返回地址多得多，保存的是用户现场。
4. 我容易忽视 `_start`，但它决定用户程序从哪里开始，以及 `main` 返回后如何退出。
5. 我容易只看组件化仓库当前文件，而忘记 Guide 中拆开的 `batch/trap/syscall` 是理解逻辑的脚手架。

## 15. 给自己讲 ch2 的推荐顺序

如果我要给别人讲 ch2，可以按下面顺序：

1. 先说 ch1 只会让内核自己打印，ch2 开始运行用户程序。
2. 再说用户程序为什么不能直接用内核资源。
3. 解释 U-mode/S-mode 和 `ecall`。
4. 解释用户程序如何在构建期被 `.incbin` 塞进内核。
5. 解释内核运行时如何找到 app 字节并复制到运行地址。
6. 解释 `_start -> main -> exit`。
7. 用 `println!` 例子讲 syscall 全链路。
8. 用 `exit` 例子讲为什么会进入下一个 app。
9. 最后对比 Guide 代码树和组件化仓库。
