# rCore ch5 补充讲述稿：从“程序能跑”到“进程可管理”

## 开场：为什么 ch5 会突然像真正的操作系统

前四章给我的感觉是：我们一直在搭运行环境。

ch1 是让裸机能启动，能输出。

ch2 是让多个应用能一个接一个跑。

ch3 是让多个应用轮流跑。

ch4 是让每个应用有独立地址空间，不会互相踩内存。

到了 ch5，系统开始像 Linux 了。因为这一章出现了进程树、父子关系、fork、exec、wait、exit。它不再只是“我写好一批程序，内核按顺序或按时间片调度它们”，而是“一个用户程序可以请求内核创建另一个用户程序，并等待它结束”。

这就是从“运行程序”走向“管理程序生命周期”。

## 面向零基础的通俗讲法

可以把每个用户程序想成一个正在办事的人。前面几章关注的是：

```text
这个人怎么进入房间？
他坐在哪里？
他的桌子和别人会不会混？
什么时候轮到他用 CPU？
```

ch5 开始关注：

```text
这个人能不能叫一个新的人来帮忙？
新的人是不是他的孩子？
孩子做完事以后谁收拾现场？
父亲要不要等孩子？
孩子能不能换一套任务继续干？
```

这就是进程管理。

## shell 的角色

我一开始会把 shell 理解成操作系统的一个神秘入口。但在 ch5 里，shell 本身就是一个用户进程。

shell 的行为类似：

```text
while true:
    读命令
    fork 一个子进程
    子进程 exec 成目标程序
    父进程 wait 子进程
    回到命令行
```

为什么 shell 不能自己 exec？因为 exec 会替换当前进程。如果 shell 自己 exec 成 `ls`，那么 ls 结束后 shell 就没了，命令行也没了。

所以 shell 要 fork 一个孩子，让孩子去 exec，自己作为父进程留下来 wait。

## 这章和 Spring Boot 的一点类比

如果用后端框架作类比，程序代码有点像 controller/service 的代码文件，进程像框架在运行期创建出来的对象实例。代码本身只是静态文本，进程才包含运行时状态。

不过 OS 的进程比普通对象更底层，它里面装的不是业务字段，而是：

- 虚拟地址空间。
- 页表。
- 寄存器上下文。
- PID。
- 父子关系。
- 调度状态。
- 堆边界。
- 退出码。

所以 ch5 的重点不是语法，而是“内核如何保存一个正在运行程序的全部状态”。

## fork：复制当前运行现场

fork 最容易让人困惑的是：为什么父子进程会从同一个位置继续执行？

因为 fork 复制的是当前进程的地址空间和上下文。所谓上下文，就是“CPU 现在执行到哪里、寄存器里有什么、栈在哪里”。复制以后，子进程看到的世界几乎和父进程一样。

差异是返回值：

```text
父进程: fork 返回 child_pid
子进程: fork 返回 0
失败: fork 返回 -1
```

于是同一份代码可以用返回值分流：

```rust
let pid = fork();
if pid == 0 {
    // 子进程逻辑
} else {
    // 父进程逻辑
}
```

这不是魔法，而是内核在创建子进程上下文时特意设置了子进程的返回寄存器。

## exec：替换当前进程身体

exec 不是创建新进程。它做的是把当前进程的用户地址空间替换掉。

可以这样理解：

```text
PID 不变
父子关系不变
但是代码段、数据段、用户栈、入口地址变了
```

所以 exec 像“换身体不换身份证”。shell 创建子进程后，子进程 exec 成目标程序，就实现了“新进程运行新程序”。

## wait：父进程回收子进程

exit 之后的进程不能立刻从系统中完全删除，因为父进程可能还要知道它的退出码。

所以内核会把退出进程标记为 Zombie。Zombie 的意思不是还在执行，而是“已经死了，但尸体还没被父进程收走”。

wait 的作用就是父进程收尸：

```text
找到已退出的子进程
读取 exit_code
删除它的内核记录
把 PID 和退出码返回给父进程
```

这就是为什么 wait 是进程生命周期里很重要的一环。

## exit：不是普通 return

用户程序 `main` 返回或者调用 `exit`，本质上都要告诉内核：这个进程结束了。

内核不能只让 CPU 停一下，而要：

- 记录退出码。
- 标记进程状态。
- 唤醒可能在 wait 的父进程。
- 处理子进程归属。
- 从调度队列里移除当前进程。

所以 exit 是一个系统状态迁移，而不是普通函数返回。

## spawn：实验题想训练什么

spawn 是本章 exercise 的新增系统调用。它可以直接创建一个运行目标程序的新进程。

它和 fork + exec 的区别是：

```text
fork + exec:
    先复制父进程
    再让子进程替换成目标程序

spawn:
    直接用目标 ELF 创建新进程
```

这也是为什么实验提示说 spawn 不必像 fork 一样复制父进程地址空间。

在实现中，spawn 最容易踩的坑是 path。用户传入的是用户虚拟地址，不是内核能直接读的地址。必须先通过当前进程的页表翻译。

## stride 调度的直觉

如果说 ch3 的调度是“排队轮流”，ch5 的 stride 调度就是“按权重轮流”。

priority 越高，应该获得越多 CPU 时间。stride 算法用一个很巧的方式实现：

```text
每个进程有 stride
每次选 stride 最小的运行
运行后 stride += BIG_STRIDE / priority
```

priority 越大，增加得越慢，下次越容易再次成为最小者，于是运行机会更多。

这比单纯轮流更公平，因为它能表达“谁更重要”。

## set_priority 的意义

`set_priority` 让用户进程自己告诉内核：我希望自己用什么优先级参与调度。

但内核不能完全信任用户输入，所以实验要求：

```text
prio < 2 -> 返回 -1
prio >= 2 -> 设置成功，返回 prio
```

这是一个很小的 syscall，但它展示了用户态影响内核调度策略的入口。

## ch5 代码树如何读

推荐阅读顺序：

```text
1. build.rs
2. src/main.rs::rust_main
3. src/process.rs::Process
4. src/processor.rs::ProcManager
5. src/main.rs::impls::Process
6. src/main.rs::impls::Memory
7. src/main.rs::impls::Scheduling
8. user/src/bin/ch5_usertest.rs
9. user/src/bin/ch5_pingpong.rs
```

不要一开始就扎进所有 unsafe 和页表细节，否则很容易迷路。先建立“谁调用谁”的链路，再看字段和数据结构。

## 本章代码调用主线

可以用下面这条线理解：

```text
用户程序
-> user_lib 封装 syscall
-> ecall
-> 内核 syscall 分发
-> main.rs::impls
-> process.rs / processor.rs
-> 修改 Process 或 ProcManager 状态
-> 返回调度循环
-> 选择下一个进程
```

这里的重点是：syscall 不只是返回一个数字，它可能改变整个系统中的进程状态。

## ch5 和 ch4 的最大区别

ch4 中的每个程序已经有地址空间，但它们更像是由内核启动时统一加载的一批程序。

ch5 中用户程序可以在运行过程中动态创建其他程序。也就是说，程序不再只是“被内核安排”，也能“请求内核安排别人”。

这就是从静态多程序系统到动态进程系统的变化。

## ch5 和 ch3 的最大区别

ch3 的任务切换主要关注：

```text
多个任务怎么轮流占用 CPU？
```

ch5 关注：

```text
任务之间有没有父子关系？
谁创建谁？
谁等待谁？
谁回收谁？
谁替换成新程序？
```

调度仍然存在，但已经不是唯一主角。进程生命周期管理成为主角。

## ch5 和真实 Linux 的对应

ch5 里的概念在 Linux 中也能看到影子：

```text
fork      -> Linux fork
exec      -> Linux execve
wait      -> Linux waitpid
exit      -> Linux exit
spawn     -> 类似 posix_spawn
priority  -> 调度权重的一种简化表达
Zombie    -> Linux 中也有僵尸进程
PID       -> Linux 进程号
```

所以这一章虽然是教学内核，但概念非常真实。

## pingpong 扩展如何讲

pingpong 扩展可以作为“进程系统 + 设备抽象”的小作品。

用户态程序做游戏逻辑：

```text
维护挡板位置
维护球位置和速度
检测碰撞
更新比分
把画面帧写给 fd=3
从 stdin 读按键
```

内核做设备支持：

```text
fd=3 -> VirtIO-GPU
stdin -> VirtIO-keyboard + UART
```

这样用户程序不需要直接碰硬件，只需要通过 read/write 系统调用请求内核服务。这正是操作系统抽象硬件的意义。

## 我在调试中学到的东西

这次 ch5 调试有三个典型问题。

第一个是 `cargo clone` 失败。它不是 Rust 代码错，而是构建脚本没找到本地用户程序目录，转而调用不存在的 cargo 子命令。解决方式是显式设置 `TG_USER_DIR`。

第二个是 `setup_framebuffer()` 失败。它不是 QEMU 没启动，也不是 ELF 错，而是 framebuffer 需要的 DMA 内存太大。800x480x4 字节约 1.46MB，DMA 池不够就失败。

第三个是图形内存不能无限加大。因为 ch5 exercise 里 spawn 会创建很多进程，内核堆也需要空间。如果图形 DMA 太大，又会挤压测试用内存。所以最后采用 640x360 + 256 页 DMA 的折中方案。
