# OS ch4 note 补充讲述稿

## 1. 从“程序放进内存”到“程序拥有地址空间”

前几章里，我更容易把程序理解为“被放到某个内存位置的一段代码”。这种理解在 ch2/ch3 还能勉强工作，因为那时还没有真正引入每个程序自己的虚拟内存世界。

到了 ch4，这个理解必须升级：

```text
程序不是简单放在某个物理地址上运行。
程序运行前，内核要为它创建一个地址空间。
地址空间里有代码段、数据段、堆、栈等区域。
这些区域对应的虚拟地址，通过页表映射到真实物理页。
```

所以 ch4 的核心不是“多了一个数据结构”，而是整个用户程序运行模型发生变化：用户看到的是虚拟地址，内核和硬件共同维护虚拟地址到物理地址的翻译。

## 2. 为什么需要页表

如果没有页表，程序访问地址 `0x10000` 就意味着访问真实物理地址 `0x10000`。这样的问题很明显：

- 多个程序容易互相覆盖。
- 用户程序可能访问内核地址。
- 内存分配必须受物理连续性的限制。
- 内核很难保护不同程序。

页表出现后，用户程序访问的 `0x10000` 只是虚拟地址。这个虚拟地址真正对应哪块物理内存，要由页表决定。

可以类比成：

```text
虚拟地址：学生看到的座位号。
物理地址：教室里真实椅子的位置。
页表：座位号到真实椅子的安排表。
内核：负责安排座位的人。
MMU：按安排表找椅子的硬件。
```

## 3. Sv39 的基本感觉

Sv39 是 RISC-V 64 位下的一种虚拟内存机制。它不是把整个虚拟地址一次性查完，而是分三级查表。

一个虚拟地址大致拆成：

```text
VPN[2] | VPN[1] | VPN[0] | page offset
```

查表流程是：

```text
satp 找到根页表
  -> 用 VPN[2] 找第二级页表
  -> 用 VPN[1] 找第一级页表
  -> 用 VPN[0] 找最终页表项
  -> 得到物理页号 PPN
  -> 拼上页内偏移 offset
  -> 得到物理地址
```

这里我需要特别记住：MMU 查页表是硬件自动做的，但页表内容是内核提前建立好的。

## 4. satp：切换地址空间的开关

`satp` 可以理解成“当前页表配置寄存器”。它告诉 CPU：

- 当前使用哪种分页模式。
- 根页表在哪里。

当内核运行某个用户程序时，需要让 CPU 使用该用户程序的页表。于是该进程的 `ForeignContext` 中不仅要保存普通寄存器，还要保存自己的 `satp`。

切换到用户程序时：

```text
内核准备好用户上下文
  -> 设置 satp 为用户进程页表
  -> sret 进入用户态
  -> 用户虚拟地址开始按该进程页表翻译
```

这就是为什么 ch4 的进程上下文比 ch3 更复杂。

## 5. Process::new 到底做了什么

`Process::new` 是 ch4 很关键的函数。它不是简单创建一个结构体，而是完成“把一个 ELF 程序变成可运行进程”的工作。

主要步骤：

```text
读取 ELF 入口地址
遍历 ELF Program Header
找到 PT_LOAD 段
为每个段分配物理页
把段内容复制进去
设置 R/W/X/U/V 权限
映射用户栈
设置用户 sp
构造 satp
返回 Process
```

ELF 中的 LOAD 段可以理解为“这个程序运行时必须被放进内存的部分”。代码段、只读数据段、可写数据段会带不同权限。

## 6. 为什么系统调用需要 translate

这是 ch4 最容易忽略，但最重要的地方。

用户程序调用：

```rust
write(fd, buf, count)
```

`buf` 是用户虚拟地址。内核收到这个数值时，不能直接：

```rust
*(buf as *const u8)
```

因为内核当前处在自己的地址空间里，这个虚拟地址对内核不一定有意义。正确做法是：

```text
找到当前进程
找到当前进程 address_space
检查 buf 所在页是否存在
检查权限是否满足读/写要求
翻译到内核可访问的物理页
再进行读写
```

所以 ch4 里的 `address_space.translate()` 是系统调用安全访问用户内存的关键。

## 7. mmap / munmap / sbrk 的意义

`mmap`、`munmap`、`sbrk` 都是在动态修改进程地址空间。

`mmap`：

```text
我要在某段虚拟地址上建立映射，并设置权限。
```

`munmap`：

```text
我要取消某段虚拟地址映射，让它之后不能再访问。
```

`sbrk`：

```text
我要扩展或收缩用户堆。
```

这些系统调用都不是单纯改变量，而是在修改页表。

## 8. ch4 Tetris 为什么适合练地址空间

俄罗斯方块看起来是游戏，但它刚好能把 ch4 的地址空间机制用起来。

用户态游戏程序负责：

- 方块生成。
- 移动和旋转。
- 碰撞检测。
- 消行。
- 计分和速度递增。
- 构造一帧 `TetrisFrame`。

内核负责：

- 从用户虚拟地址读取 `TetrisFrame`。
- 初始化 VirtIO-GPU。
- 把棋盘数据渲染成像素。
- 读取 VirtIO-keyboard 输入。

这说明一个图形游戏应用并不是“用户程序直接操作硬件”，而是：

```text
用户程序表达需求
内核检查和翻译用户数据
内核代表用户访问硬件
```

## 9. 本次调试中最关键的 bug

最开始我看到的问题是：QEMU 窗口出来了，但是没有画面。

后来定位过程发现：

```text
用户程序已经启动
用户程序也在提交图形帧
但是内核访问 VirtIO-GPU MMIO 地址时触发 LoadPageFault
```

原因是 ch4 开启了页表。页表开启后，内核访问设备地址也必须先映射。之前内核地址空间只映射了内核代码、数据、堆和传送门，没有映射 QEMU 的 MMIO 设备区。

修复思路：

```text
在 kernel_space() 中映射 0x1000_0000..0x1000_3000
其中包括 UART、VirtIO-GPU、VirtIO-keyboard
```

修复后日志出现：

```text
virtio-gpu ready
virtio-keyboard ready
```

这说明设备访问链路真正通了。

## 10. 给别人讲这一章时可以这样说

如果要向别人解释 ch4，我会按这个顺序讲：

1. 前几章程序已经能运行，但内存还没有真正隔离。
2. ch4 引入虚拟地址和物理地址的分离。
3. 每个进程都有自己的页表，也就是自己的地址空间。
4. `satp` 指向当前页表，MMU 根据页表做地址翻译。
5. 加载 ELF 时，内核把代码段、数据段、用户栈映射进用户地址空间。
6. 系统调用访问用户指针时，必须用 `translate()`。
7. `mmap/munmap/sbrk` 本质上是在改页表。
8. 做 Tetris 时，用户态提交 frame，内核翻译用户指针并驱动 GPU。
9. 因为开了页表，内核访问 MMIO 设备也必须映射设备地址。
10. 所以 ch4 的主线是：地址空间隔离不仅保护内存，也影响系统调用和设备驱动。
