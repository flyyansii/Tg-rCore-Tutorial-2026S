# OS ch4 note 补充讲述稿

## 1. 从“程序放进内存”到“程序拥有地址空间”

前几章里，我更容易把程序理解为“被放到某个内存位置的一段代码”。这种理解在 ch2/ch3 还能勉强工作，因为那时还没有真正引入每个程序自己的虚拟内存世界。

到了 ch4，这个理解必须升级：

```text
程序不是简单放在某个物理地址上运行。
程序运行前，内核要为它创建一个地址空间。
地址空间里有代码段、数据段、堆、栈等区域。
这些区域对应的虚拟地址，通过页表映射到真实物理页。
```

所以 ch4 的核心不是“多了一个数据结构”，而是整个用户程序运行模型发生变化：用户看到的是虚拟地址，内核和硬件共同维护虚拟地址到物理地址的翻译。

## 2. 为什么需要页表

如果没有页表，程序访问地址 `0x10000` 就意味着访问真实物理地址 `0x10000`。这样的问题很明显：

- 多个程序容易互相覆盖。
- 用户程序可能访问内核地址。
- 内存分配必须受物理连续性的限制。
- 内核很难保护不同程序。

页表出现后，用户程序访问的 `0x10000` 只是虚拟地址。这个虚拟地址真正对应哪块物理内存，要由页表决定。

可以类比成：

```text
虚拟地址：学生看到的座位号。
物理地址：教室里真实椅子的位置。
页表：座位号到真实椅子的安排表。
内核：负责安排座位的人。
MMU：按安排表找椅子的硬件。
```

## 3. Sv39 的基本感觉

Sv39 是 RISC-V 64 位下的一种虚拟内存机制。它不是把整个虚拟地址一次性查完，而是分三级查表。

一个虚拟地址大致拆成：

```text
VPN[2] | VPN[1] | VPN[0] | page offset
```

查表流程是：

```text
satp 找到根页表
  -> 用 VPN[2] 找第二级页表
  -> 用 VPN[1] 找第一级页表
  -> 用 VPN[0] 找最终页表项
  -> 得到物理页号 PPN
  -> 拼上页内偏移 offset
  -> 得到物理地址
```

这里我需要特别记住：MMU 查页表是硬件自动做的，但页表内容是内核提前建立好的。

## 4. satp：切换地址空间的开关

`satp` 可以理解成“当前页表配置寄存器”。它告诉 CPU：

- 当前使用哪种分页模式。
- 根页表在哪里。

当内核运行某个用户程序时，需要让 CPU 使用该用户程序的页表。于是该进程的 `ForeignContext` 中不仅要保存普通寄存器，还要保存自己的 `satp`。

切换到用户程序时：

```text
内核准备好用户上下文
  -> 设置 satp 为用户进程页表
  -> sret 进入用户态
  -> 用户虚拟地址开始按该进程页表翻译
```

这就是为什么 ch4 的进程上下文比 ch3 更复杂。

## 5. Process::new 到底做了什么

`Process::new` 是 ch4 很关键的函数。它不是简单创建一个结构体，而是完成“把一个 ELF 程序变成可运行进程”的工作。

主要步骤：

```text
读取 ELF 入口地址
遍历 ELF Program Header
找到 PT_LOAD 段
为每个段分配物理页
把段内容复制进去
设置 R/W/X/U/V 权限
映射用户栈
设置用户 sp
构造 satp
返回 Process
```

ELF 中的 LOAD 段可以理解为“这个程序运行时必须被放进内存的部分”。代码段、只读数据段、可写数据段会带不同权限。

## 6. 为什么系统调用需要 translate

这是 ch4 最容易忽略，但最重要的地方。

用户程序调用：

```rust
write(fd, buf, count)
```

`buf` 是用户虚拟地址。内核收到这个数值时，不能直接：

```rust
*(buf as *const u8)
```

因为内核当前处在自己的地址空间里，这个虚拟地址对内核不一定有意义。正确做法是：

```text
找到当前进程
找到当前进程 address_space
检查 buf 所在页是否存在
检查权限是否满足读/写要求
翻译到内核可访问的物理页
再进行读写
```

所以 ch4 里的 `address_space.translate()` 是系统调用安全访问用户内存的关键。

## 7. mmap / munmap / sbrk 的意义

`mmap`、`munmap`、`sbrk` 都是在动态修改进程地址空间。

`mmap`：

```text
我要在某段虚拟地址上建立映射，并设置权限。
```

`munmap`：

```text
我要取消某段虚拟地址映射，让它之后不能再访问。
```

`sbrk`：

```text
我要扩展或收缩用户堆。
```

这些系统调用都不是单纯改变量，而是在修改页表。

## 8. ch4 Tetris 为什么适合练地址空间

俄罗斯方块看起来是游戏，但它刚好能把 ch4 的地址空间机制用起来。

用户态游戏程序负责：

- 方块生成。
- 移动和旋转。
- 碰撞检测。
- 消行。
- 计分和速度递增。
- 构造一帧 `TetrisFrame`。

内核负责：

- 从用户虚拟地址读取 `TetrisFrame`。
- 初始化 VirtIO-GPU。
- 把棋盘数据渲染成像素。
- 读取 VirtIO-keyboard 输入。

这说明一个图形游戏应用并不是“用户程序直接操作硬件”，而是：

```text
用户程序表达需求
内核检查和翻译用户数据
内核代表用户访问硬件
```

## 9. 本次调试中最关键的 bug

最开始我看到的问题是：QEMU 窗口出来了，但是没有画面。

后来定位过程发现：

```text
用户程序已经启动
用户程序也在提交图形帧
但是内核访问 VirtIO-GPU MMIO 地址时触发 LoadPageFault
```

原因是 ch4 开启了页表。页表开启后，内核访问设备地址也必须先映射。之前内核地址空间只映射了内核代码、数据、堆和传送门，没有映射 QEMU 的 MMIO 设备区。

修复思路：

```text
在 kernel_space() 中映射 0x1000_0000..0x1000_3000
其中包括 UART、VirtIO-GPU、VirtIO-keyboard
```

修复后日志出现：

```text
virtio-gpu ready
virtio-keyboard ready
```

这说明设备访问链路真正通了。

## 10. 给别人讲这一章时可以这样说

如果要向别人解释 ch4，我会按这个顺序讲：

1. 前几章程序已经能运行，但内存还没有真正隔离。
2. ch4 引入虚拟地址和物理地址的分离。
3. 每个进程都有自己的页表，也就是自己的地址空间。
4. `satp` 指向当前页表，MMU 根据页表做地址翻译。
5. 加载 ELF 时，内核把代码段、数据段、用户栈映射进用户地址空间。
6. 系统调用访问用户指针时，必须用 `translate()`。
7. `mmap/munmap/sbrk` 本质上是在改页表。
8. 做 Tetris 时，用户态提交 frame，内核翻译用户指针并驱动 GPU。
9. 因为开了页表，内核访问 MMIO 设备也必须映射设备地址。
10. 所以 ch4 的主线是：地址空间隔离不仅保护内存，也影响系统调用和设备驱动。

## 11. 讲述用流程：从“虚拟地址”展开成 29 步

如果要把 ch4 讲给别人，我会这样从虚拟地址开始讲：

1. 前三章用户程序已经能跑，但内存模型还比较直接。
2. ch4 开始，用户程序不再直接面对物理内存。
3. 用户程序看到的是虚拟地址。
4. 每个进程都有自己的虚拟地址空间。
5. 虚拟地址空间由页表描述。
6. 页表项记录虚拟页到物理页的映射。
7. 页表项还记录权限，例如 R/W/X/U。
8. `satp` 告诉 CPU 当前根页表在哪里。
9. CPU 访问虚拟地址时，MMU 自动查页表。
10. 查表成功后得到物理地址。
11. 查表失败或权限不对就触发 PageFault。
12. 内核加载 ELF 时，要按 ELF 段建立映射。
13. 代码段通常可读可执行。
14. 数据段通常可读可写。
15. 只读段通常可读不可写。
16. `.bss` 虽然文件里没有内容，但运行时要有清零内存。
17. 用户栈也要映射。
18. 堆区域通过 `sbrk` 扩展。
19. `mmap` 可以建立新的用户虚拟内存区域。
20. `munmap` 可以撤销映射。
21. 用户程序传给内核的指针是虚拟地址。
22. 内核处理 syscall 时必须先翻译用户指针。
23. 翻译时必须检查权限。
24. 如果用户传坏地址，内核不能直接崩，要返回错误或杀掉用户程序。
25. Tetris 里用户传给内核的 frame 指针也要翻译。
26. 内核解析 frame 后再驱动 GPU。
27. 键盘输入也是内核读设备、用户读抽象输入。
28. 开启分页后，MMIO 地址也必须在内核页表中映射。
29. ch4 的核心就是：地址空间成为连接程序加载、syscall、内存保护和设备驱动的基础结构。

## 12. Tetris 实验如何反过来帮助理解 ch4

俄罗斯方块不是单纯“做个游戏”，它把 ch4 的地址空间机制暴露得很明显：

1. 游戏逻辑在用户态。
2. 用户态只维护棋盘、方块、分数、速度。
3. 用户态不能直接写 framebuffer。
4. 用户态把一帧画面整理成 `TetrisFrame`。
5. 用户态通过 `write(fd=3)` 把 frame 交给内核。
6. frame 指针是用户虚拟地址。
7. 内核必须通过当前进程的页表翻译 frame 指针。
8. 内核确认 frame 可读后，才能解析。
9. 内核把逻辑格子转换为像素。
10. 内核写 VirtIO-GPU framebuffer。
11. GPU flush 后，QEMU 显示窗口更新。
12. 用户按键由 VirtIO-keyboard 进入内核。
13. 内核把硬件按键转换成简单字符。
14. 用户态通过 `read(STDIN)` 获取字符。
15. 这说明用户态和设备之间始终隔着 syscall 和地址翻译。

## 13. 按 Guide 模块讲 ch4：每个文件为什么存在

ch4 的 Guide 不是随便把代码拆成很多文件，而是按“地址空间从底到顶”的顺序拆开的。

### `address.rs`

这一层最底。它把普通数字地址变成有语义的类型：

```text
虚拟地址 VirtAddr
物理地址 PhysAddr
虚拟页号 VirtPageNum
物理页号 PhysPageNum
```

我可以把它理解成：先把“地址”这件事说清楚，后面才能说页表。

### `frame_allocator.rs`

页表映射必须落到真实内存，所以内核要知道哪些物理页还空着。这个模块就是物理页帧管理员。

```text
用户代码段需要物理页。
用户数据段需要物理页。
用户栈需要物理页。
页表中间节点也需要物理页。
```

### `page_table.rs`

这一层开始真正处理 Sv39 页表。它负责：

```text
创建页表项 PTE。
建立 VPN -> PPN 映射。
删除映射。
查询虚拟地址对应的物理地址。
检查权限。
```

### `memory_set.rs`

单独 map 一个页还不够。一个进程需要一整个地址空间，包括代码、数据、栈、堆、mmap 区域。`MemorySet` 就是“一个进程的虚拟内存世界”。

组件化版本里，这些能力主要被封装进：

```text
tg-kernel-vm::AddressSpace
tg-kernel-vm::space::mapper
tg-kernel-vm::space::visitor
tg-kernel-alloc
```

## 14. 讲述用模块调用链：ch4 不是单模块，而是一串联动

```text
main.rs
  -> 初始化内核
  -> 创建 kernel_space
  -> 开启页表

kernel-vm
  -> 提供 AddressSpace
  -> 提供 map/translate 能力

process.rs
  -> 读取用户 ELF
  -> 调用 AddressSpace 映射 LOAD 段
  -> 保存进程自己的地址空间

main.rs syscall 实现
  -> 根据 caller 找当前进程
  -> 根据进程 AddressSpace 翻译用户指针
  -> 执行 write/read/mmap/munmap/sbrk

graphics.rs
  -> 接收翻译后的用户 frame
  -> 操作 VirtIO-GPU

keyboard.rs
  -> 从 VirtIO-keyboard 读取事件
  -> 转换成用户态可理解的按键
```

这条链的关键是：`AddressSpace` 不只在进程创建时用一次，后面每次系统调用涉及用户指针时都要用。

## 15. ch4 和 Guide 主线的一句话对应

```text
Guide 讲 mm/address/page_table/memory_set，
本质是在教我：用户程序看到的地址不是物理现实，而是内核为它布置出来的一套虚拟世界。
```

而组件化仓库是在这个思想上更进一步：

```text
把地址空间能力封装成 crate，
让 ch4/ch5/ch6 后续章节都能复用同一套虚拟内存基础。
```
